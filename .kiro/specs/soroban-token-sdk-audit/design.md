# Design Document: soroban-token-sdk Audit

## Overview

The `token-factory` contract incorrectly uses `TokenClient` from `soroban-token-sdk` to make cross-contract mint and burn calls. `soroban-token-sdk` is a helper library for *implementing* SEP-41 token contracts — it is not designed for calling external token contracts. The correct approach is `token::Client` from `soroban-sdk`, which is the official cross-contract client for SEP-41 tokens.

This design covers the two-line fix in `lib.rs`, the corresponding `Cargo.toml` cleanup, and the testing strategy to verify correctness.

## Architecture

No architectural changes are required. The TokenFactory contract structure, storage layout, event emissions, and public API remain identical. The only change is the internal implementation of two cross-contract calls:

- `mint_tokens`: replace `TokenClient::new(...).mint(...)` → `token::Client::new(...).mint(...)`
- `burn`: replace `TokenClient::new(...).burn(...)` → `token::Client::new(...).burn(...)`

The `use soroban_token_sdk::TokenClient;` import is removed. The `token` module is already imported via `soroban_sdk` in the existing `use` statement.

```
Before:
  use soroban_token_sdk::TokenClient;
  ...
  TokenClient::new(&env, &token_address).mint(&admin, &to, &amount);
  TokenClient::new(&env, &token_address).burn(&from, &amount);

After:
  (no soroban_token_sdk import)
  ...
  token::Client::new(&env, &token_address).mint(&admin, &to, &amount);
  token::Client::new(&env, &token_address).burn(&from, &amount);
```

## Components and Interfaces

### token::Client (soroban-sdk)

`token::Client` is generated by `soroban-sdk` and implements the SEP-41 token interface for cross-contract calls. Relevant methods:

```rust
// Mint `amount` tokens to `to`, authorized by `admin`
fn mint(env: &Env, admin: &Address, to: &Address, amount: &i128);

// Burn `amount` tokens from `from`
fn burn(env: &Env, from: &Address, amount: &i128);
```

Both signatures match the existing call sites in `mint_tokens` and `burn`, so no argument changes are needed.

### Cargo.toml Changes

`contracts/token-factory/Cargo.toml` requires three removals:

1. `[dependencies]`: remove `soroban-token-sdk = { version = "21.0.0" }`
2. `[dev-dependencies]`: remove `soroban-token-sdk = { version = "21.0.0", features = ["testutils"] }`
3. `[features]`: remove `soroban-token-sdk/testutils` from the `testutils` feature list

## Data Models

No data model changes. `TokenInfo`, `FactoryState`, and `Error` are unchanged.

## Correctness Properties

*A property is a characteristic or behavior that should hold true across all valid executions of a system — essentially, a formal statement about what the system should do. Properties serve as the bridge between human-readable specifications and machine-verifiable correctness guarantees.*

### Property 1: Cross-contract dispatch uses token::Client

*For any* valid token address, when `mint_tokens` or `burn` is called on the TokenFactory, the call SHALL be dispatched to the target token contract using `token::Client` from `soroban-sdk`, and the target contract SHALL receive the correct arguments (admin/from, to, amount).

**Validates: Requirements 2.1, 2.2**

### Property 2: Source is free of soroban-token-sdk references

*For any* build of the TokenFactory, the compiled source SHALL contain no references to `soroban_token_sdk` or `TokenClient` from that crate.

**Validates: Requirements 2.3, 2.4**

### Property 3: Cargo manifest is clean

*For any* inspection of `contracts/token-factory/Cargo.toml`, the manifest SHALL contain no entries for `soroban-token-sdk` in `[dependencies]`, `[dev-dependencies]`, or `[features]`.

**Validates: Requirements 3.1, 3.2, 3.3**

## Error Handling

No new error conditions are introduced. The existing `Error` enum and all error paths in `mint_tokens` and `burn` are unchanged. `token::Client` raises panics (contract errors) on failure in the same way `TokenClient` did, so error propagation behavior is preserved.

## Testing Strategy

### Dual Testing Approach

**Unit tests** (existing `test.rs`): The existing tests cover pause/unpause behavior and fee validation. They do not exercise the actual cross-contract mint/burn paths (those calls fail in the unit test environment because no real token contract is deployed). These tests should continue to pass unchanged after the replacement.

**Build verification**: `cargo build` is the primary correctness gate — a successful build confirms the import replacement is syntactically and semantically correct and that `soroban-token-sdk` is fully removed.

**Test execution**: `cargo test` verifies no behavioral regressions were introduced.

### Property-Based Testing

This feature is a targeted refactor with no algorithmic logic changes. The correctness properties (Property 1, 2, 3) are verified through:

- **Property 1**: Integration test using `soroban-sdk`'s `testutils` to register a mock SEP-41 token contract and assert that `mint_tokens` and `burn` dispatch calls to it correctly.
- **Property 2**: Static check — `grep` / `rg` confirms no `soroban_token_sdk` references remain in `lib.rs`.
- **Property 3**: Static check — `Cargo.toml` contains no `soroban-token-sdk` entries.

Property-based testing library: `proptest` (already present in `Cargo.toml`) can be used for generating random `Address` and `i128` values in integration tests if needed.

**Tag format**: `Feature: soroban-token-sdk-audit, Property {N}: {property_text}`

Each property test must run a minimum of 100 iterations when using randomized inputs.
